# KAIKO　06. マネタイズ設計

ver 1.0　|　2026年3月

---

## 1. 基本方針

初期フェーズはユーザー数の獲得・データの蓄積を最優先とし、課金による離脱リスクを避ける。収益化は段階的に導入し、蓄積されたデータ・コミュニティ・信頼性を資産として活用する構造を目指す。

| 項目 | 方針 |
| --- | --- |
| 初期スタンス | 基本機能は無料提供。まずユーザーが使い続ける体験を作る |
| 課金タイミング | Phase 3以降。データと信頼が蓄積されてから段階導入 |
| 課金の原則 | 回数制限より情報の深度制限。価値を体験してから判断できる設計 |
| 避けること | ユーザーデータの無断販売・UXを損なう広告過多 |

## 2. 収益モデル一覧

以下の8施策を検討する。現フェーズのスコープは06（アフィリエイト）のみ。他はすべて将来構想として位置づける。

| No. | 施策名 | 概要・KAIKOとの親和性 | タイミング |
| --- | --- | --- | --- |
| 01 | 競馬関連広告運用 | 競馬コンテンツに特化した広告枠。単価は高いが、コア層への過剰な広告表示はUXを損なうリスクあり。補助的な収益源として位置づける | 将来 |
| 02 | 予想マーケットプレイス | 予想精度の高いユーザーが自分の予想を販売できる場を提供し、プラットフォームが手数料を得る。ユーザー自身が販売主体となるため法的にクリーン。競技化機能との連動で自然発生する | 将来 |
| 03 | コンテンツ販売（学習教材） | 熟練ユーザーの予想手法・回顧プロセスを体系化した教材・コンテンツを販売。02の発展形として整理できる | 将来 |
| 04 | 深度制限による機能課金 | 基本の回顧閲覧は無料、物差し馬比較・詳細分析・AIレポートなど高度機能を有料化。回数制限ではなく情報の深さで課金ラインを引く | Phase 3〜 |
| 05 | APIアクセス権の提供 | 蓄積した構造化回顧データを外部ツール・研究者・競馬アプリにAPI提供。精査済み回顧データは競合が持たない独自資産になりうる | 将来 |
| 06 | アフィリエイト提携 | 予想ソフト・競馬新聞・馬券購入サービスへの送客で収益化。UX破壊が少なく競馬特化ターゲットで単価が高い。初期から導入可能な唯一の収益源 | 早期〜 |
| 07 | プライベートリーグ（有料） | 友人・仲間内でクローズドな予想リーグを作れる機能。グループ内でのランキング・成績比較・コメント共有。競技化の延長線上にあり、口コミによるバイラル拡散が期待できる | Phase 3〜 |
| 08 | AIパーソナル分析レポート | ユーザーの蓄積データをもとに予想の強み・弱み・高回収率条件をAIが分析。G1前などの特定タイミングで都度課金。データが蓄積されるほど価値が上がるループを形成する | Phase 3〜 |

## 3. 収益化ロードマップ

| フェーズ | 収益施策・方針 |
| --- | --- |
| Phase 1〜2（現在〜近期） | 06（アフィリエイト）のみ導入。それ以外は無料提供に徹し、ユーザー獲得とデータ蓄積を優先する |
| Phase 3〜（中期） | 04（深度制限課金）・07（プライベートリーグ）・08（AIレポート）を段階導入。ユーザーが価値を体験済みの状態で課金を提示する |
| 将来（長期） | 02・03（マーケットプレイス・コンテンツ販売）・05（API提供）・01（広告）を本格展開。データとコミュニティを資産として活用するフェーズ |

## 4. 重点施策の補足

### 08. AIパーソナル分析レポート（推奨施策）

8施策の中で最もKAIKOのコア価値と直結している。ユーザーが回顧・予想を蓄積するほどレポートの精度が上がり、「使い続けるほど自分専用のツールになる」体験を設計できる。G1シーズンごとの都度課金モデルは、継続的な収益サイクルとしても機能する。

### 07. プライベートリーグ（バイラル起点）

競馬は友人・仲間内で予想を競い合う文化が強い。クローズドな予想リーグ機能は「LINEグループの競馬版」として使われ、招待経由でのユーザー流入が期待できる。課金理由として「みんなで使う体験」を提供できる唯一の施策。

### 02. 予想マーケットプレイス（競技化との連動）

6章の競技化機能と直接連動する。ランキング上位ユーザーが自分の予想を販売できる仕組みにすることで、「精度を上げるほど収益機会が生まれる」インセンティブ構造が完成する。プラットフォームが手数料を得る設計とすることで、ユーザーデータの無断販売を回避しながら収益化できる。

> ※ 本ドキュメントはすべて検討段階の構想を含む。具体的な課金額・機能仕様・実施タイミングは別途設計する。
